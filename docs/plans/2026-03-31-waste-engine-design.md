## 1. Problem Statement

Large supermarket chains lose millions annually to perishable waste across produce, bakery, dairy, meat, and ready meals. Current systems apply static markdown schedules and react to waste after it occurs. This system predicts waste risk at SKU-store-batch level and autonomously selects optimal actions — discounts, inter-store transfers, or replenishment adjustments — before waste occurs.

---

## 2. Chosen Approach

**Pure ADK Multi-Agent Pipeline + Gradio UI**

Four specialized ADK agents orchestrated by a SequentialAgent, backed by synthetic supermarket data, with a Gradio frontend for demo. Deployed to Vertex AI Agent Engine and Cloud Run.

Chosen over notebook-first (less agentic) and Vertex AI Agent Builder (less technically deep) approaches.

---

## 3. Agent Architecture

```
User Trigger (Gradio UI)
        ↓
OrchestratorAgent  [ADK SequentialAgent]
        ↓
DataAgent
  └── tool: load_inventory(store_id, sku_id)
  └── Returns: at-risk batches with expiry, quantity, velocity
        ↓
ForecastAgent
  └── tool: score_spoilage_risk(batch)
  └── Gemini 1.5 Flash prompt → risk score (0–1) + reason string
        ↓
DecisionAgent
  └── tool: simulate_actions(batch)
  └── Simulates 4 scenarios: no_change / 10%_discount / 20%_discount / transfer
  └── Selects action maximising: waste_reduction × margin_protection
        ↓
ExplanationAgent
  └── tool: generate_explanation(decision)
  └── Gemini 1.5 Flash → plain English "why" narrative
  └── Returns structured decision card for Gradio
```

Each agent is a standalone ADK `Agent` class with registered tools (Python functions). The OrchestratorAgent uses `SequentialAgent` to chain them and passes shared state via ADK's session context.

## 4. Data Layer

**Synthetic data generated by `generate_data.py` (run once on Day 1 morning):**

| File | Columns | Rows |
|------|---------|------|
| `inventory.csv` | sku_id, store_id, batch_id, quantity, expiry_date, cost_price, sell_price, days_to_expiry | ~500 |
| `sales_history.csv` | sku_id, store_id, date, units_sold, day_of_week, weather_tag | ~2000 |
| `stores.csv` | store_id, name, location, region, size | 5 stores |
| `skus.csv` | sku_id, name, category, shelf_life_days, avg_daily_velocity | ~50 SKUs |

Weather tags: `hot / cold / normal / rainy` — simulated, no API needed.
Hero demo scenario: Store A, 120 packs chicken, 2 days to expiry, declining velocity, heatwave ending.

## 7. Gradio UI Layout

**3-panel layout:**

- **Panel 1 — Inventory Explorer:** Store selector dropdown + SKU table with colour-coded expiry risk (red / amber / green)
- **Panel 2 — Risk Heatmap:** Bar chart of waste risk scores per SKU batch
- **Panel 3 — Agent Decision Output:** Decision card (action selected, waste prevented, margin impact) + Gemini explanation text

**Bonus widgets:**
- ESG counter: "Waste prevented today: X units / £Y saved"
- Gemini chat box: ask "Why did you choose transfer over discount?"
